## 二、数据同步方案

### 方案对比

| 维度 | Firebase Realtime DB | Supabase | 手动 JSON 同步 |
|---|---|---|---|
| 实时性 | ✅ 毫秒级 | ✅ WebSocket | ❌ 手动 |
| 离线支持 | ✅ SDK 内建 | ⚠️ 需自行实现 | ❌ 无 |
| 免费额度 | 1GB / 10万连接/月 | 500MB / 2项目 | 无限 |
| SDK 体积 | ~80KB gzipped | ~30KB | 0 |
| 中国可用性 | ⚠️ 需梯子（用户已确认可用）| ✅ 有亚洲节点 | ✅ |

### 选型结论：Firebase（用户已确认有梯子）

> v1→v2 变更（来源：🔴1）：审查指出应评估 Firebase 中国可用性。经用户确认有梯子，Firebase 方案可行。
> 如未来需要去梯子环境使用，可降级回 localStorage + 手动导出/导入（V1 方案），数据不丢失。

**降级策略**：Firebase 不可用时自动降级为 localStorage 模式（代码层面做 try-catch）。

### 实现方案

```
Firebase Auth（Google 登录 — 一键登录，无需注册）
  ↓
Firebase Realtime Database（JSON 存储，实时监听）
  ↓
离线模式：enablePersistence() → 本地缓存 → 上线自动同步
  ↓
V1 数据迁移：首次登录时检测 localStorage → 提示一键上传
```

## 三、UI 设计分析（v1→v2 重写，来源：🔴2 — 从场景倒推）

### 3.1 用户实际使用场景分析

> v1→v2 变更：v1 从设计趋势出发列举 Glassmorphism 等方向，审查指出应从用户场景倒推。以下按使用频率排序。

**场景 A：训练中记录（每周 3 次，每次 15+ 交互，最高频）**
- 环境：健身房，可能嘈杂
- 身体状态：组间休息 90-120 秒，手可能出汗
- 操作需求：看到上次重量 → 输入本次重量+次数 → 下一个动作
- UI 需求：**大触控区域、高对比度、数字键盘优先、单手可操作**
- ❌ 不适合：细腻的毛玻璃效果（汗手+强光下可读性差）

**场景 B：早起记录体重（每天 1 次，30 秒，中频）**
- 环境：家里，安静
- 操作需求：输入体重 → 看到趋势
- UI 需求：**大输入框、快速完成、趋势图一目了然**
- ✅ 适合：精美的数据可视化

**场景 C：仪表盘概览（每天 1-2 次，只看不操作，中频）**
- 环境：任意
- 操作需求：一眼看到今天的状态（训练了没/蛋白够了没/体重趋势）
- UI 需求：**信息密度高但不拥挤、数字突出、颜色编码**
- ✅ 适合：Glassmorphism 卡片、圆形进度环、渐变和动效

**场景 D：周复盘（每周 1 次，5 分钟，低频）**
- 环境：家里，安静
- UI 需求：自动预填、减少输入
- ✅ 适合：丰富的数据展示

### 3.2 场景驱动的 UI 原则

| 原则 | 说明 | 来源场景 |
|---|---|---|
| **输入区高对比度实色**  | 训练记录的输入区域用实色背景+大字号，不用半透明 | 场景 A |
| **展示区可以炫酷** | 仪表盘、趋势图等只看区域用 Glassmorphism + 动效 | 场景 B/C |
| **数字优先** | 核心数据用 40-64px 醒目字号 | 场景 A/B/C |
| **触控友好** | 最小触控区 48×48px，按钮间距 ≥ 8px | 场景 A |
| **单手可操作** | 关键操作在屏幕下半部分 | 场景 A |
| **3 秒录入** | 一组训练数据录入 < 3 秒（参考 Strong） | 场景 A |

### 3.3 V2 视觉方向

基于场景分析，V2 采用**双区设计**：

| 区域 | 设计风格 | 适用 |
|---|---|---|
| **展示区**（仪表盘/图表/进度） | Glassmorphism + 渐变 + 圆形进度环 + 微动画 | 概览、趋势、成就 |
| **操作区**（训练记录/输入表单） | 高对比度实色 + 大按钮 + 数字键盘 | 训练中快速录入 |

**色彩体系**：
- 背景：深色渐变（#0a0a1a → #1a1035）
- 卡片：毛玻璃效果（rgba(255,255,255,0.05) + backdrop-filter: blur(20px)）
- 强调色系统：蓝 #4f7df7 / 紫 #7c3aed / 绿 #10b981 / 橙 #f59e0b
- 进度环三色：体重进度 蓝 / 训练完成 绿 / 蛋白达标 紫

## 四、反面教材分析（v1→v2 新增，来源：🔴3）

### 为什么 GitHub 上的健身追踪项目大多停更？

调研了 20+ 个开源健身追踪项目，归纳失败模式：

| 失败模式 | 频率 | 典型案例 | 我们如何避免 |
|---|---|---|---|
| **技术栈过重** | 高 | React+Redux+TypeScript+Node+MongoDB | 保持纯 HTML/CSS/JS + Firebase CDN |
| **功能膨胀** | 高 | 第一版就做社交/AI/视频等 | MVP 只做核心追踪 + 同步 |
| **没有真实用户** | 高 | 做给 Portfolio 而非自用 | 我们是真实用户（Dogfooding） |
| **移动端适配差** | 中 | 只适配桌面 Web | 移动优先设计 |
| **部署太复杂** | 中 | 需要自建 Docker+PostgreSQL | Firebase Hosting 一键部署 |

### wger Workout Manager 为什么增长慢？

- ✅ 功能全面（训练+饮食+体重，开源 7 年）
- ❌ UI 极度过时（Bootstrap 3 风格，2015年设计感）
- ❌ 移动端体验差（响应式但不是移动优先）
- ❌ 安装要搭 Django 后端 + PostgreSQL
- **教训**：功能全 ≠ 用户愿意用。**UI 质量和部署方便性是个人工具的生死线**

## 五、V2 方向结论

### 数据互通

| 方面 | 方案 |
|---|---|
| 后端 | Firebase Realtime Database（CDN 引入）|
| 认证 | Firebase Auth（Google 登录）|
| 离线 | Firebase Persistence + localStorage 降级 |
| 迁移 | V1 localStorage 数据一键上传（v1→v2 详化，来源：🟡2）|
| 部署 | Firebase Hosting 或 GitHub Pages（v1→v2 新增，来源：🟡3）|
| 状态 | UI 显示同步状态（✅ 已同步 / ⟳ 同步中 / 📴 离线）|

### UI 升级

| 方面 | V1 现状 | V2 方向 |
|---|---|---|
| 卡片 | 实色 + 边框 | **展示区**毛玻璃 / **操作区**高对比实色 |
| 图表 | CSS 柱状图 | SVG 圆形进度环 + CSS 柱状图 |
| 进度 | 线性进度条 | 三环进度环（体重/训练/饮食） |
| 动效 | fadeIn | 数字 count-up + 进度环动画 + 按钮波纹 |
| 背景 | 纯色 #0a0a1a | 渐变 #0a0a1a → #1a1035 |

### 功能增量

| 功能 | 优先级 |
|---|---|
| 数据实时同步 | 🔴 必做 |
| UI 全面升级（双区设计） | 🔴 必做 |
| 围度追踪 | 🟡 应做 |
| 同步状态指示器 | 🟡 应做 |
| V1→V2 数据迁移 | 🟡 应做 |
| 训练进步曲线 | 🟢 可选 |
