# 通用踩坑预警库

> **适用**：所有项目类型
> **维护方式**：每次项目经验沉淀（Phase 5）时追加新条目

---

## 🔴 P0: 初始化逻辑必须覆盖所有状态分支

```
场景：初始化/启动逻辑只处理了"正常路径"，遗漏了异常分支
根因：开发时只跑了 happy path，没有模拟各种初始状态
修复：画出状态机全部分支，确保每个分支的末尾都有一个确定的后续操作
预防：在编码完成后，列出所有可能的初始状态组合，逐一验证
来源：fitness Phase 5.5 白屏 bug（Firebase 已配置+未登录→不渲染）
```

## 🟡 P1: 外部依赖版本号必须在线查证

```
场景：凭记忆写了第三方库/SDK 的版本号，实际已过时
根因：人的记忆不可靠，版本号变化快
修复：每次引用外部依赖时，去官方网站/仓库查最新稳定版
预防：代码中使用外部依赖时，在注释中标注版本查证日期
来源：fitness Phase 5.5 Firebase SDK 版本从 v9.23.0 改为 v12.10.0
```

## 🟡 P2: 每个外部服务调用必须有 try-catch + 降级

```
场景：外部服务（API/数据库/CDN）不可用时，整个应用崩溃
根因：代码假设外部服务永远可用
修复：try-catch 包裹每个外部调用，失败时执行降级策略
预防：设计阶段就定义每个外部依赖的降级方案
来源：fitness Phase 5.5 Firebase 降级为 localStorage
```

## 🟡 P3: 本地测试通过 ≠ 部署环境通过

```
场景：本地运行正常，部署后出现意外行为
根因：本地环境（file://、localhost）掩盖了部署环境（HTTPS、CDN）的路径/协议差异
修复：部署后必须在目标环境再验收一次
预防：开发时模拟目标环境的关键约束（如 HTTPS only、跨域限制）
来源：fitness Phase 5.5 OAuth 需要 HTTPS
```

## 🟢 P4: Phase 间产出物必须被下游消费

```
场景：前一个 Phase 产出了完整文档，但下一个 Phase 没有引用
根因：Phase 之间缺少价值链检查
修复：每个 Phase 启动时确认"前序 Phase 产出了什么？我如何使用？"
预防：Phase 间价值链检查（已内建于 full-cycle-builder 流程）
来源：agent-os Phase Value Chain 设计
```
