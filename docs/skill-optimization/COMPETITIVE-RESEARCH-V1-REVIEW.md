## 🔴 结构性问题（3 个）

### 问题 1：缺少真实场景验证——不同项目类型的执行路径差异分析

**诊断**：调研对比了 7 个竞品方案，但全部停留在方法论/框架层面。没有验证关键假设——**"同一套 5 Phase 流程能否适配 Web/后端/CLI/数据分析等不同项目类型？"**

**后果**：如果不同项目类型的差异仅在"踩坑库内容"层面，那分类化就够了。但如果差异在"Phase 结构本身"——比如 CLI 工具不需要部署阶段、数据分析项目的验收标准完全不同——那 5 Phase 结构就需要可选跳过机制。

**建议**：新增一节"项目类型执行路径分析"，至少覆盖 3 类项目：
1. Web 前端应用（已有 fitness 案例）
2. 工具链/框架项目（已有 agent-os 案例）
3. 非代码项目（文档/方案/调研型项目）

逐类分析：哪些 Phase 必做？哪些可跳过？质量门标准有何不同？

---

### 问题 2：引力陷阱过滤不够深——Appetite 机制和 Skeleton-First 被标为 🔴 必做，但风险评估不够

**诊断**：从 Shape Up 借鉴的 Appetite 机制被标为 🔴 必做，但没有分析：
- Appetite 和现有质量门是否矛盾？（时间到了但质量门没过怎么办？）
- 在 AI Agent 环境下，"时间 budget" 的含义是什么？（Agent 不受工时限制）

**后果**：Appetite 机制可能引入新的复杂度，与现有的"质量门驱动"逻辑产生冲突。

**建议**：
- 分析 Appetite 和质量门的兼容性
- 明确"时间 budget"在 AI Agent 环境下的是谁的时间？是用户的时间（等待 Agent 完成的时间），还是项目的生命周期预算？
- 如果冲突，选择保留哪个：质量门优先 or 时间预算优先？

---

### 问题 3：缺少"内部案例回溯"——两个项目的实际执行偏差分析

**诊断**：调研提到了 "来源案例" 但只是一句话引用。没有深入分析：当 fitness-muscle-building 和 agent-os 实际执行时，**哪些步骤与 full-cycle-builder 定义的流程产生了偏差？偏差的原因是什么？**

**后果**：不分析实际偏差，优化就是"理论改理论"，而不是"从实践中提炼"。

**建议**：新增"内部案例回溯"节：
- fitness 项目：实际走了 Phase 1→2→3→5→5.5，但 Full-cycle-builder 是后来才写的。如果从头用 full-cycle-builder 走，哪些步骤对不上？
- agent-os 项目：走的是 project-planner 的 8 Phase 流程，与 full-cycle-builder 的 5 Phase 有什么映射关系？

## 🟡 设计盲点（4 个）

### 盲点 1：Skill 结构方案 B 的 references/ 子文件加载机制未说明

**缺什么**：技术选型结论选了方案 B（SKILL.md + references/），但没有说明 Agent 如何"按项目类型加载对应文件"。
**为什么重要**：不同 Agent（Claude/Gemini/Codex）对 references/ 目录的自动加载行为不同。如果 Agent 不自动读取 references/，分类化就失去意义。
**建议**：说明 references/ 的触发机制——是在 SKILL.md 中显式指令"请读取 references/pitfall-library/web.md"，还是靠 Agent 自动发现？

### 盲点 2：调研缺少对 project-planner 现有流程的兼容性分析

**缺什么**：project-planner 已经定义了完整的 Phase 1-5 流程（信息获取→对接→初始化→审查→沉淀），full-cycle-builder 也定义了 Phase 0-5。两者如何共存？
**为什么重要**：如果两个 Skill 的 Phase 结构冲突，用户不知道该调哪个。
**建议**：增加一节分析 project-planner 和 full-cycle-builder 的 Phase 对应关系，明确互斥边界。

### 盲点 3：未评估 code-review Skill 的当前状态

**缺什么**：差距分析提到"缺少 code-review 环节"，但没有检查 code-review Skill 当前是否可用、API 是什么。
**为什么重要**：如果 code-review Skill 不存在或不成熟，就不能在优化中把它列为必调用的子 Skill。
**建议**：确认 code-review Skill 的当前状态，再决定是否纳入。

### 盲点 4：未分析 WORKFLOW-fitness-plan.md 中已有的经验沉淀

**缺什么**：fitness 项目有一份完整的工作流文档 `WORKFLOW-fitness-plan.md`（15857 字节），里面可能已经沉淀了 full-cycle-builder 可以吸收的模式。
**为什么重要**：WORKFLOW 文档是实际经验的第一手记录，比调研对比更有参考价值。
**建议**：阅读 WORKFLOW-fitness-plan.md，提取其中未被当前 SKILL.md 覆盖的模式。

## 🟢 优化建议（2 个）

### 建议 1：竞品对比矩阵增加"实现复杂度"维度

当前矩阵从功能角度对比，但缺少"引入该方案的改造成本"估算。比如引入 Appetite 机制只需加几行判断逻辑，但引入项目类型判定可能需要整个 references/ 目录。

### 建议 2：调研报告末尾增加机器可读摘要

按 deep-research Skill 的标准，报告末尾应有 YAML 摘要，方便后续 Skill 链消费。
