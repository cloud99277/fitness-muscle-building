## 🔴 结构性问题（3 个）

### 问题 1：违反「每个 Phase 只做 1 件事」—— 10 个模块不是 1 件事

**诊断**：PHASE-5.md 包含 10 个任务（T0~T10），覆盖 7 个功能模块 + 数据层 + 设计系统 + 验收。这不是一个 Phase，这是一个包含多个 Phase 的子项目。

**后果**：
- 开发时间远超预期（手写 Canvas 图表 + 7 个交互模块，纯 JS 乐观估计也要 2000+ 行代码）
- 无法在合理时间内交付可用产品
- 违反工作流的核心节奏约束

**建议**：将 Phase 5 拆为 MVP 和迭代两部分：

**MVP（Phase 5 核心交付物）**：
- T0: 项目初始化 + 数据层
- T2: UI 设计系统（CSS 变量 + 基础样式）
- T3: 仪表盘（核心概览）
- T4: 体重记录（最核心的追踪）
- T5: 训练记录（使用频率最高）
- T10: 验收

**后续迭代（Phase 5.5 或 6）**：
- T6: 饮食检查
- T7: 周复盘
- T8: 问题诊断
- T9: 里程碑

---

### 问题 2：手写 Canvas 图表是过度工程

**诊断**：v1 明确要求「Canvas API 手写，不引入 Chart.js 等第三方库」来绘制体重趋势图、训练重量曲线等。手写 Canvas 图表需要处理坐标轴、刻度、缩放、触摸事件、响应式等，代码量大且容易出 bug。

**后果**：
- 手写一个可用的折线图组件需要 300-500 行 JS，调试时间至少占总开发时间的 30%
- 结果还是不如现成库好看
- 「零外部依赖」的收益远小于「花一天时间手写一个残缺的图表库」的成本

**建议**：两个方案任选：
1. **方案 A（推荐）**：用纯 CSS 实现简化图表（CSS Grid 柱状图 + CSS 数据可视化），不走 Canvas
2. **方案 B**：用 CDN 引入超轻量的图表库（如 Chart.js 仅 60KB gzipped），CDN 引入和 Google Fonts 一样，都是外部 CDN，不算 npm 依赖

---

### 问题 3：缺少 MVP 验收标准与核心用户场景

**诊断**：验收标准列出了 7 个通用检查项，但没有回答最关键的问题：**用户每天实际使用时的操作流程是什么？**

**后果**：如果不定义核心用户场景，开发出来的 app 可能符合所有技术指标但「用起来不顺手」。

**建议**：新增「核心用户场景」，定义 3 个高频场景的端到端操作流：

1. **训练日场景**（每周 3 次）：
   打开 app → 选训练日类型 A → 逐动作记录重量和次数 → 保存 → 看到「本周已训练 2/3」

2. **每日体重场景**（每天早起）：
   打开 app → 输入今日体重 → 看到趋势图 → 看到「距离目标还差 12kg」

3. **周五复盘场景**（每周 1 次）：
   打开 app → 点「周复盘」→ 看到自动预填的本周数据 → 确认或修改 → 保存

## 🟡 设计盲点（4 个）

### 盲点 1：文件结构过度拆分

**缺什么**：7 个模块分成 7 个独立 JS 文件（modules/ 目录下），对于纯 HTML 项目来说加载 10+ 个 JS 文件会导致模块间依赖管理困难（无 import/export），需要手动管理全局变量和加载顺序。
**为什么重要**：没有打包工具时，模块间通信只能靠全局变量或事件总线，复杂度快速上升。
**建议**：合并为 3 个文件——`store.js`（数据层）+ `components.js`（所有 UI 组件）+ `app.js`（路由和初始化），减少文件间依赖。

### 盲点 2：数据导出是关键路径但描述太简略

**缺什么**：v1 提到「导出 JSON / 导入 JSON」但只有一句话，没有具体设计。
**为什么重要**：localStorage 是单浏览器存储，换手机 / 清缓存 = 数据丢失。对于持续 8-12 个月的追踪项目来说，数据丢失是灾难级风险。
**建议**：
- 导出：点击按钮 → 生成带日期的 JSON 文件 → 自动下载
- 导入：选择文件 → 预览数据量 → 确认覆盖 / 合并
- 自动备份提醒：每周日弹窗「建议导出备份」

### 盲点 3：未定义 SPA 路由方案

**缺什么**：v1 提到 `app.js` 负责「路由」，但没有说明单页应用的路由方案是什么。
**为什么重要**：纯 HTML 项目没有 React Router 等工具，需要自己实现页面切换。
**建议**：用 hash 路由（`#/dashboard`, `#/weight`, `#/training` 等）+ `hashchange` 事件监听，是最简单的 SPA 方案。每个「页面」是一个 `<section>` 元素，通过 CSS display 切换显隐。

### 盲点 4：训练模板数据应与 V2 方案保持同步

**缺什么**：v1 列出了 A/B/C 训练模板的动作，但没有说明这些数据的来源是硬编码还是可配置。
**为什么重要**：如果用户后续升级训练方案（从全身训练到上下肢分化），需要能修改模板。
**建议**：训练模板存储在 `store.js` 的默认数据中，用户可以在设置页面修改或重置。

## 🟢 优化建议（3 个）

### 建议 1：增加数据示例模式
开发完后内置一套 30 天的示例数据，让用户开箱就能看到图表和趋势，降低「空白页面」的挫败感。

### 建议 2：底部 Tab 导航减为 4 个
5 个 Tab 在小屏手机上偏挤。建议 4 个：仪表盘 / 记录（体重+围度） / 训练 / 更多（饮食+复盘+诊断+设置）。

### 建议 3：使用 emoji 代替自定义 SVG 图标
对于个人工具，用 emoji（💪🏋️📊⚡）代替自定义 SVG 图标，省去 assets/icons/ 目录的维护成本。
