### Phase 1-4 决策（方案制定阶段）

| 决策 | 为什么这么做 | 效果 |
|---|---|---|
| 先读工作流规范再建项目结构 | 避免随意建文件，后续难以管理 | ✅ 项目结构清晰，有 PROJECT.md 锚点 |

| 决策 | 为什么这么做 | 效果 |
|---|---|---|
| 双循环工作流（调研→审查 + 规划→审查） | 调研和规划是不同维度，分开迭代质量更高 | ✅ 两个循环独立收敛 |

## 可复用模式

### Phase 1-4 模式（方案迭代通用）

```
1. V1 是好的起点，不是终点
   - 任何专业方案第一稿都有结构性盲点
   - 「我写得挺好的」和「用架构师视角找问题」是两种思维

2. 最容易被忽略的维度
   - 调整机制：当计划不 work 时怎么办？（比计划本身更重要）
   - 第一步指导：用户的第一个行动是否有具体指引？
   - 高频场景：用户真实环境和理想环境的差距在哪？

3. 审查的分层模型依然普适
   - 🔴 结构性：不修会导致整体失效
   - 🟡 盲点：不影响启动但实施前要回答
   - 🟢 优化：锦上添花
   - 收敛标准：全部🟡时通过，不求完美

4. 「执行系统」vs「说明书」
   - 说明书：告诉你该怎么做
   - 执行系统：告诉你该怎么做 + 遇到障碍怎么处理 + 如何判断效果
   - 面向零经验用户必须是执行系统
```

### Phase 5/5.5 模式（App 开发通用）★ 新增

```
5. 双循环工作流 — 调研和规划分开迭代
   - 循环 1：调研 → 审查 → 修订 → 再审查（收敛「做什么」）
   - 循环 2：规划 → 审查 → 修订 → 再审查 → 执行（收敛「怎么做」）
   - 两个循环独立运转，不能混在一起
   - 调研审查发现的问题不同于规划审查：引力陷阱 vs 执行矛盾

6. 初始渲染兜底 — Web App 的致命陷阱
   - 认证状态有多个分支：未配置/已配置未登录/已登录
   - 每个分支都必须触发 UI 渲染，否则白屏
   - init() 的最后一行应该是兜底 render
   - 教训：本地测试时「未配置→离线模式」路径遮盖了「已配置未登录」路径
   - 根因：测试覆盖的用户路径不完整

7. CDN vs npm — 零 npm 项目的取舍框架
   - CDN 优点：零构建、file:// 可用、改完刷新生效
   - CDN 缺点：版本号需手动查、无 tree-shaking、无 TypeScript
   - 决策线：个人工具 + <10 文件 + 不需要构建 → CDN；否则 npm
   - 关键纪律：写 CDN 版本号时必须查最新版本，不凭记忆

8. 降级策略 — 外部服务不可靠时的兜底
   - Firebase 可用 → 实时同步（主路径）
   - Firebase 不可用 → localStorage（降级路径）
   - UI 必须显示当前模式（「已同步」/「离线」）
   - try-catch 包裹每个外部调用，不让网络错误崩溃整个 App
   - Store API 对上层保持一致，降级只在内部切换

9. 场景驱动的双区 UI — 不是所有页面都应该炫
   - 展示区（仪表盘/趋势图）→ Glassmorphism/动效/渐变（引发成就感）
   - 操作区（训练记录/数字输入）→ 高对比度/大按钮/实色背景（健身房可用）
   - 场景决定设计，不是风格决定设计
   - inputmode="decimal" 在手机上弹数字键盘，比 type="number" 体验更好

10. 部署闭环 — 代码写完不等于完成
    - 写完代码 → 本地验收 → 部署到 HTTPS → 填入外部服务配置 → 真机验收
    - Google 登录等 OAuth 必须 HTTPS，file:// 不行
    - GitHub Pages + Actions workflow 是零 npm 项目的最佳部署方案
    - 部署后必须在目标环境（手机浏览器）再验收一次
```

## 踩坑记录 ★ 新增

### 🔴 白屏 Bug（Phase 5.5，部署后发现）

```
根因：app.js init() 中 Firebase 三个认证状态只覆盖了两个

    ┌─ Firebase 未配置 → initFirebase() 返回 false → 兜底 render ✅
    │
    ├─ Firebase 已配置 + 已登录 → onAuthChange(user) → render ✅
    │
    └─ Firebase 已配置 + 未登录 → onAuthChange(null) → if(user) 不满足 → 不 render ❌ 白屏！

修复：在 onAuthChange 回调中加 else 分支，未登录也渲染登录页
教训：
  1. 认证状态是状态机，必须覆盖所有分支
  2. 本地测试时 Firebase 未配置（走第一条路径），遮盖了第三条路径
  3. 应该在本地也测试「填入配置但不登录」这个场景
  4. init() 方法应该在所有分支的最后都有一个确定的 render 调用
```

### 🟡 SDK 版本错位（Phase 5.5，配置时发现）

```
根因：凭记忆写了 Firebase SDK v9.23.0，但 Firebase Console 给出的最新版是 v12.10.0
影响：功能无影响（compat 向后兼容），但不规范
教训：写第三方 CDN 链接时，必须查最新版本号，不能凭记忆
```

## 一句话总结

> V1 方案是科学的「说明书」，V2 方案是可独立执行的「执行系统」。
> V1 App 是单机 MVP，V2 App 是多设备同步的高级工具。
> 全过程五个关键杠杆：**审查分层迭代**（每轮提升 2 分），**双循环工作流**（调研和规划独立收敛），**场景驱动 UI**（展示区炫+操作区实用），**降级策略**（外部依赖 always fail），**部署闭环**（代码写完≠完成）。
> 唯一一个上线后才发现的 🔴 bug（白屏）来自认证状态机覆盖不全——**你以为测过了的路径，不代表所有路径都测过了。**
